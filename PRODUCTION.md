# Lumora — Production Checklist

## Backend (Go API)
- [ ] Copy `backend/.env.example` → `backend/.env` and fill real values.
- [ ] **JWT_SECRET**: set to a long random value (`openssl rand -hex 32`).
- [ ] **CORS_ORIGINS**: set to your frontend HTTPS origin (e.g. `https://your-domain.com`).
- [ ] **APP_URL**: your frontend HTTPS URL (used in emails + Paystack callback + password-reset links).
- [ ] **SMTP_***: real mailbox (Gmail App Password works; a domain mailbox via SendGrid/Mailgun/SES is better for deliverability).
- [ ] **PAYSTACK_SECRET_KEY / PAYSTACK_PUBLIC_KEY**: switch to **live** keys.
- [ ] In the Paystack dashboard set the **Webhook URL** to `https://api.your-domain.com/api/paystack/webhook` (needed so payments are fulfilled even if the user closes the tab).
- [ ] Serve over **HTTPS** (required for camera + screen share in the proctored exam, and for secure cookies/webhooks).
- [ ] Consider Postgres instead of SQLite for concurrent production load (swap the GORM driver).

## Deploying the backend to Render (`render.yaml`)
- [ ] Dashboard → **New → Blueprint** → select this repo. The blueprint defines the
      `lumora-api` web service, its disk and its env vars.
- [ ] **Plan must be paid (`starter`).** The disk holds SQLite + avatar uploads, and
      Render only allows disks on paid instances. A free instance has an ephemeral
      filesystem: every deploy and idle-spindown wipes all user data.
- [ ] `DB_PATH` and `UPLOADS_DIR` must stay under the disk's `mountPath` (`/var/data`).
- [ ] `JWT_SECRET` is auto-generated by Render. Rotating it logs every user out.
- [ ] Fill the `sync: false` vars in the dashboard: `CORS_ORIGINS`, `APP_URL`, SMTP, Paystack.
- [ ] Set `CORS_ORIGINS` + `APP_URL` to the exact Netlify origin, **no trailing slash**.
- [ ] Point the Paystack webhook at `https://<service>.onrender.com/api/paystack/webhook`.
- [ ] First request after idle is slow (cold start) — expected on Render.

## Deploying the frontend to Netlify (`netlify.toml`)
- [ ] Add a new site from this repo; `netlify.toml` sets `base = "frontend"` and the
      Next.js runtime plugin, so no manual build settings are needed.
- [ ] `output: "standalone"` must stay **out** of `next.config.js` — it's incompatible
      with Netlify's runtime and fails the build.
- [ ] Set `NEXT_PUBLIC_API_URL` (the Render URL) and `NEXT_PUBLIC_SITE_URL` in
      Site configuration → Environment variables **before the first build**. These are
      inlined at build time, so changing them later needs a redeploy.
- [ ] Favicon/logo is `app/icon.svg`; PWA manifest at `/manifest.webmanifest`; OG image `public/logo.svg`.

### Chicken-and-egg on the URLs
The two hosts reference each other, so expect a second pass:
1. Deploy Render → note the API URL.
2. Deploy Netlify with `NEXT_PUBLIC_API_URL` set → note the site URL.
3. Set `CORS_ORIGINS` + `APP_URL` on Render to the Netlify URL → Render redeploys.

## Security
- [ ] `.env` files are gitignored — never commit real keys.
- [ ] HTTPS everywhere; set `CORS_ORIGINS` to the exact frontend origin.
- [ ] Rotate any keys that were used in development/testing.

## Smoke test after deploy
- [ ] Sign up → welcome email arrives.
- [ ] Log in → sign-in email; app loads.
- [ ] Forgot password → reset email → set new password → log in.
- [ ] Buy an exam attempt (live/test) → receipt email + exam unlocks.
- [ ] Lose hearts → regenerate/purchase works; out-of-hearts ends a lesson.
- [ ] Take an exam → proctoring (camera + screen) prompts; certificate issues with the stamp.
